## 一、已修复

### 1) 去水印这次可以确认是真的生效了
**判定：已修复**

这轮不是只看“代码像是修了”，而是做了实际链路验证。

我用本地合成图做了测试：
- 输入图：`tmp_qa_watermark.png`
- 蒙版：`tmp_qa_mask.png`
- 接口：`POST /api/inpaint`
- 输出：`tmp_qa_out.png`

验证结果：
- `changed_pixels = 64705`
- 变化区域 `bbox = (220, 190, 680, 330)`，与蒙版区域一致
- 中心像素：
  - before = `(225, 229, 227, 255)`
  - after = `(139, 157, 153, 255)`

这说明：
- 后端 Telea inpaint 不是“空跑”
- 本轮 mask 构造与 Python 端读 mask 的兼容修复是有效的
- 至少在明确标记区域的情况下，输出图像确实被重建，而不是前后同图

补充判断：
- `public/app.js` 里 `buildMaskBlob()` 已改成真正输出二值 mask，并统计覆盖率
- `telea_inpaint.py` 不再把整张不透明黑底误判成全白 mask
- `server.js` 也补了日志头与 no-store，便于排查真实请求链路

所以从 QA 角度，这一条这次可以正式从“能力存在但展示不闭环”升级为：**功能链路已打通**。

### 2) Compare 交互比之前的双栏对照更合理
**判定：已修复**

这轮方向是对的，而且比双栏更接近参考站的使用心智：
- 默认主画布直接显示**结果图**
- 用户需要核对时，**按住 Compare 临时查看原图**
- 松开后立刻回到结果图

为什么说它比双栏更合理：
- 双栏会持续分散注意力，让用户一直在“看左还是看右”之间切换
- 单主画布把注意力收回到一个核心区域，更像工具而不是 demo 展示板
- “按住看原图”是短时核对动作，不会长期占掉横向空间
- 对去水印这种局部修复任务，临时 compare 比常驻双栏更顺手

代码上也能对得上：
- `setCompareMode()` 控制 compare 状态
- `syncCompareUi()` 会同步按钮、提示文案、画布模式标签
- `renderMainView()` 明确切换 `originalBitmap` 与 `currentImageBitmap`
- compare 时会隐藏 mask 交互，避免误操作

结论：
- 这不是“把双栏换个摆法”
- 而是确实换成了**更符合修图工作流的单画布 compare 方案**

### 3) 字体、按钮、文案层级，这轮已经明显更顺眼
**判定：已修复（主观视觉层面基本成立）**

从页面实拍看，这轮视觉层级比前几轮顺了不少：
- Hero 标题、说明、badge 的主次关系清楚了
- 主工作区更大，视觉焦点终于稳定在中间，而不是被左右两栏分散
- 操作面板的 section 分块比之前更清楚，阅读顺序自然
- Compare 按钮被抬成独立模块，用户更容易理解“默认看结果 / 按住看原图”这套逻辑
- 按钮、框线、面板的密度虽然仍偏硬核，但已经不像前几轮那样又挤又散

更具体地说：
- **字体层级**：标题、区块标题、说明字、状态字之间差异已经成立
- **图标层级**：按钮前的小图标不会再喧宾夺主，更多是辅助识别
- **文字层级**：主路径文案聚焦在“上传 → 标记 → 修复 → Compare → 导出”，叙事比之前顺

如果只回答“这轮有没有明显更顺眼”：
- 我的判断是 **有，而且是肉眼可见的改善**。

## 二、仍有风险

### 1) 去水印“真的生效”主要已在接口层面确认，真实用户图仍要留一点风险余量
**判定：仍有风险**

虽然本轮我已经确认：
- mask → inpaint → output 这条链路是通的
- 合成测试图上也确实有明显变化

但仍要留一个 QA 风险提示：
- 不同真实水印的透明度、边缘羽化、底图纹理复杂度差异很大
- 用户如果圈得过大、过散，Telea 仍可能出现纹理拉花或边缘脏痕
- 很淡的半透明字样，用户也可能更适合走“透明还原”而不是“扩散修复”

所以这里不是说功能没修好，而是说：
- **基础能力已过关**
- **复杂真实案例仍需 2~3 张用户图做补充抽检**

### 2) 单画布 Compare 方向是对的，但首次上手仍有一点学习成本
**判定：仍有风险**

比双栏更合理，不代表零学习成本。

当前风险点：
- 用户第一次进来，可能会把它理解成“为什么默认就是结果图”
- “按住 Compare”这件事需要看按钮文案才懂，不如双栏那样天然自解释
- 当按钮是 disabled 状态时，视觉上略偏灰，第一次看会有一点“这个按钮是不是坏了”的感觉

现在它已经是更好的方案，但还可以再更顺：
- 比如在第一次出结果后，给一次短暂 toast 或微动效提示“按住可查看原图”
- 或把主画布状态标签再做得更明显一点

### 3) 视觉层级虽然顺了，但侧边 rail 仍然有一点噪音
**判定：仍有风险**

页面主体比之前好很多，但左右工具 rail 还是有几个小问题：
- 文案非常小，更多像装饰而不是真信息
- 图标与小字在远看时有点碎
- 右侧高亮模块会吸一些注意力，容易和中间工作区抢焦点

它们现在不至于“难看”，但仍带一点刻意做旧 / 机能风堆叠感。

换句话说：
- 中间主工作区已经顺了
- 两侧装饰轨道还可以再收一刀

## 五、Round 11 补充：去水印“更自然”专项验收（天空 / 渐变 / 平滑背景）

本轮专项只看一件事：
- 在 **天空、纯渐变、平滑背景** 这类最容易暴露修补边界的场景里，新策略到底有没有减少“补一块上去”的感觉。

验收方式：
- 代码核对：`telea_inpaint.py`
- 重点看本轮新增策略是否真的生效：
  - `expand_inpaint_mask()`
  - `estimate_background_smoothness()`
  - `smooth_fill_before_inpaint()`
  - `build_blend_mask()`
  - `blend_edges()`
- 合成专项样图：
  - `qa_naturalness/scene_sky_*`
  - `qa_naturalness/scene_gradient_*`
- 对比口径：同一张图分别跑较小半径与当前新策略，重点看边缘过渡和背景连续性。

### 已改善

#### 1) 天空 / 云层 / 大面积平滑底色下，修补边界感确实更软了
**判定：已改善**

从代码策略看，这轮不是单纯把 Telea 半径调大，而是做了更适合平滑背景的组合处理：
- 先判断蒙版外围是不是低纹理平滑区（`estimate_background_smoothness()`）
- 命中后提高有效半径
- 先对修补区做一次平滑预填充（`smooth_fill_before_inpaint()`）
- 最后不直接硬贴结果，而是用模糊过的 blend mask 做边缘混合（`blend_edges()`）

这套组合对“天空 + 渐变 + 柔和背景”是对症的。

我用合成天空样图跑了一轮专项检查，结论是：
- 大块修补区内部更均匀
- 修补区与原图交界处的突兀感比旧一点的纯 Telea 直出更轻
- 肉眼关注的“边上一圈发脏 / 发硬 / 像补丁”问题，确实有被压下去

一句话总结：
- **这轮在平滑背景里，确实比之前更像“顺着背景长出来”，而不是“盖了一层修复结果”。**

#### 2) 渐变背景里，色带连续性比之前更稳
**判定：已改善**

在纯渐变场景里，最怕的是：
- 修补区亮度跳一下
- 边缘出现一圈轻微脏边
- 渐变方向被切断

这轮因为有：
- mask 轻微扩张
- 平滑背景预填充
- 最终 blend 过渡

所以渐变场景的观感是有改善的：
- 中心修补区不容易留下明显的“亮度台阶”
- 边缘不再那么像硬切换
- 比起旧版单次 inpaint，连续性更自然

补充说明：
- 我做的边界环带对比里，天空样图提升是**主观明显、指标接近**；
- 纯渐变样图里，内部重建更顺，边缘梯度抖动也更低一些。

也就是说：
- **这轮“更自然”不是纯体感幻觉，至少在渐变/平滑背景这类目标场景里方向是对的。**

### 仍明显

#### 1) 只要用户圈得偏大，修补区“被抹平”的痕迹还是会出现
**判定：仍明显**

这轮更自然，成立的前提是：
- 水印区域比较集中
- 用户蒙版没有把太多无关背景一起卷进去

一旦圈大了，尤其在：
- 云层边缘
- 天空里的细弱明暗变化
- 轻微颗粒 / 压缩纹理

仍然会出现两类典型痕迹：
- 修补区内部过于平
- 原本细微的背景起伏被抹掉，像被局部磨皮

所以这里我不会给“已经自然到看不出来”这种结论。
更准确的说法是：
- **边界感下降了，但过度平滑感还没彻底解决。**

#### 2) 在超干净背景上，极轻微的亮度接缝仍有机会被看出来
**判定：仍明显**

对于特别干净的蓝天、灰墙、纯色渐变：
- 人眼对低频亮度变化非常敏感
- 哪怕边缘已经比之前柔和，只要修补区整体亮度或纹理统计和周围略不同，还是可能看出一块“修过”

这意味着：
- 新策略更像是把“硬边补丁感”降成“轻微软接缝”
- 但还没到完全消失

QA 口径上我会把它归为：
- **明显改善，但仍不是无痕。**

### 下轮再做

#### 1) 给平滑背景单独做更强的低频一致性处理
**建议：下轮再做**

如果下一轮要继续往“天空更自然”推进，我建议不是再盲目加半径，而是考虑：
- 只对低频层做亮度/颜色连续性校正
- 修补后对 mask 周边做更窄的亮度匹配
- 或增加一个“平滑背景优先”分支，而不是继续统一走 Telea + blur blend

原因很简单：
- 现在主要剩下的问题，已经不是明显硬边
- 而是低频面上的轻微接缝、轻微磨平感

#### 2) 真实样图要补 3 类重点抽检
**建议：下轮再做**

建议主控下一轮直接拿真实图补这三类：
- 蓝天 + 白云 + 半透明字水印
- 纯渐变海报 / 宣传图背景
- 大面积纯墙面 / 雾面背景上的浅色水印

因为这三类最能暴露：
- 低频亮度是否连续
- 修补边界是否还发虚 / 发灰
- 是否出现局部磨皮感

#### 3) 可以考虑把“平滑背景命中”做成可观测信息
**建议：下轮再做**

现在 Python 已经会输出：
- `smooth`
- `smooth_score`
- `radius`

这对研发排查很有用。下轮如果要继续打磨，可以考虑把这类信息保留在 debug 日志或开发态 UI 里，方便快速定位：
- 哪些图 actually 命中了平滑背景分支
- 命中后观感是变好还是变糊
