## B. 当前仍存在的部署风险

### 1. `render.yaml` 与文档口径仍不完全一致
- 风险级别：高
- 当前现象：
  - `render.yaml` 中写的是 `rootDir: img-toolbox-app`
  - 但 `DEPLOYMENT.md` / `README.md` 里多处写的是 **Root Directory 留空（仓库根目录）**
- 为什么会误导：
  - 当前工作区里 `img-toolbox-app/` 自己就是一个独立 repo；如果用户直接把这个 repo 接到 Render，`rootDir: img-toolbox-app` 会变成错误路径
  - 也就是说：**Blueprint 适用于“父仓库包含子目录”的场景，文档则更像“当前目录就是仓库根”的场景**
- 建议：
  - 二选一，统一口径
  - 如果当前 GitHub 仓库根就是 `img-toolbox-app`，那 `render.yaml` 应去掉 `rootDir`
  - 如果确实要保留子目录仓库结构，则 README / DEPLOYMENT 需要明确写出“这是 monorepo 子目录部署”

### 2. `vercel.json` 只写了 `npm run prepare:frontend`，与文档中的 build 口径不完全一致
- 风险级别：中
- 当前现象：
  - `vercel.json`：`buildCommand` 是 `npm run prepare:frontend`
  - `DEPLOYMENT.md`：示例写法是 `npm install && npm run prepare:frontend`
- 影响：
  - 实际上 Vercel 通常会先自动安装依赖，这一版大概率能跑
  - 但对照文档时会让人怀疑“到底要不要手写 npm install”
- 建议：
  - 文档里明确说明：Vercel 默认会自动安装依赖，因此 `vercel.json` 里只保留 `npm run prepare:frontend` 即可
  - 或者把两边写法统一成同一套表达

### 3. 仓库内默认 `public/runtime-config.js` 仍是空字符串，若线上环境变量漏配，前端会静默退回同源 `/api`
- 风险级别：高
- 当前现象：
  - `public/runtime-config.js` 当前仍是：`apiBaseUrl: ""`
  - `app.js` 中 `buildApiUrl()` 在空值时会直接走同源 `/api/inpaint`
- 为什么会误导：
  - 这保证了本地一体运行可用，但也意味着 **Vercel 若忘了配置 `IMGEXE_API_BASE_URL`，页面还能打开，只是请求会悄悄打到 Vercel 同源 `/api/inpaint`，然后失败**
  - 用户看到的是“前端能打开但功能不通”，不够显错
- 建议：
  - 在前端启动时增加显式提示：当域名不是 localhost 且 `apiBaseUrl` 为空时，展示“未配置后端 API 地址”的状态警告
  - 或在 Vercel 构建阶段对生产环境空值直接 fail-fast

### 4. Render Blueprint 环境变量还没把跨域白名单一起模板化
- 风险级别：中
- 当前现象：
  - `server.js` 已支持 `CORS_ALLOW_ORIGINS`
  - 但 `render.yaml` 里未示例化这项 env
- 影响：
  - 现在默认相当于放开，短期利于打通
  - 但切生产后若想收紧来源，需要手工补；文档与模板没完全闭环
- 建议：
  - 在 `render.yaml` 中补一个可选注释示例，或在文档中强调“上线后应把 Vercel 域名写入 `CORS_ALLOW_ORIGINS`”

### 5. 部署文档虽然更清楚了，但“当前推荐架构”与“本地一体运行”两种模式仍容易被初次部署者混淆
- 风险级别：中
- 当前现象：
  - README 同时讲了：
    - 本地 `npm start` 一体运行
    - 线上 Vercel + Render 分离部署
  - 对熟悉工程的人没问题，但对第一次部署的人，仍可能分不清“什么时候应该看到空 `apiBaseUrl`，什么时候一定要填线上地址”
- 建议：
  - 在 README 顶部增加一个 3 行结论块：
    - 本地开发：`apiBaseUrl` 可以为空
    - Vercel 线上：`IMGEXE_API_BASE_URL` 必填
    - Render：只负责 `/api/*`

## C. 当前 UI 对照后仍存在的主要偏差

### 1. 整体方向对了，但浅色主题与参考站默认深色主题仍不一致
- 偏差级别：高
- 当前对照：
  - 参考站默认是黑底、强工业感、设备壳体感明显
  - 本地截图当前处于浅色 `INK` 主题，整体像“反相版 clone”
- 影响：
  - 结构已经很像，但第一眼气质仍不完全像参考站
- 建议：
  - 若目标是“默认打开就像参考站”，默认主题应改回 dark
  - light 主题可保留为可切换功能，而不是默认首屏

### 2. 顶栏已经更接近，但仍比参考站更“宽松、更精致”，少一点粗粝设备味
- 偏差级别：中
- 现象：
  - 当前顶栏字距、盒子、分隔线已经接近
  - 但本地版顶部留白仍更均匀，边框/字重更像精修过的 web clone
  - 参考站更硬、更糙、更像旧式桌面工具外壳
- 建议：
  - 继续压缩顶栏局部留白
  - 弱化一些“太干净”的对称感，让信息块更像工具站壳体而不是设计稿

### 3. 中央 hero strip 仍然存在，参考站则几乎是“标题条后直接进入工作台”
- 偏差级别：高
- 现象：
  - 当前有 `hero-strip`，含英文说明、中文说明、右侧 badge
  - 参考站中央标题条之后，几乎立即进入黑色工作区 + 控制面板
- 影响：
  - 虽然现在已经比前几轮短很多，但这块仍然会把页面拉得更像“介绍型产品页”
- 建议：
  - 若要继续贴参考站，可把 `hero-strip` 再压薄，甚至直接并入 panel 顶条

### 4. 左侧工作区空态文案与按钮仍偏“居中展示页”，参考站更克制
- 偏差级别：中
- 现象：
  - 当前空态的「拖拽图片到工作台 / 选择图片」视觉权重较大
  - 参考站空态更像工作台默认占位，上传提示更低调
- 建议：
  - 缩小空态标题字号
  - 降低主按钮面积与高亮强度
  - 提高“这是编辑器默认底板”而不是“这是一个 CTA 区”的感觉

### 5. 右侧控制栏结构接近了，但统计卡 + 多层边框 still 稍显“组件化”
- 偏差级别：中
- 现象：
  - 当前状态/标记/输出 三卡已经压缩很多
  - 但仍带有明显的现代组件栅格感
  - 参考站右栏更像一整块连续设备面板，而不是由多个卡片拼装
- 建议：
  - 继续减少子卡片边界感
  - 提升连续竖向控制流，而不是分明的卡片盒子感

### 6. 右 rail 激活态已经更像，但左右 rail 图标语义和参考站仍非一一对应
- 偏差级别：低
- 现象：
  - 视觉结构已接近
  - 个别图标、排列和参考站并非逐项同构
- 影响：
  - 不影响主要相似度，也不影响功能
- 建议：
  - 若目标是“截图级复刻”，可以继续细调；若目标是“高相似但不逐像素”，这项可降优先级

### 7. 页脚已补齐，但文案仍与参考站不完全一致
- 偏差级别：低
- 现象：
  - 当前为 `© 2026 IMGEXE TOOLBOX · POWERED BY DOINGFB`
  - 参考站更接近 `© 2026 图像处理工具箱 · POWERED BY DOINGFB`
- 影响：
  - 完成度已够，但细节一致性还差半步

## E. 本轮 QA 结论

### 部署结论
- 这轮修复后，**部署思路已经比上一轮清楚很多**：
  - 前端静态资源收敛到 `public/`
  - 运行时 API 配置抽到 `runtime-config.js`
  - Vercel / Render 的职责边界已经建立起来
- 但还没有到“完全不会误导”的程度：
  - `render.yaml` 与文档中的 Root Directory 口径仍冲突
  - `apiBaseUrl` 为空时的生产环境失败模式仍偏静默

### UI 结论
- 这轮前端确实**明显更接近参考站**，特别是：
  - 三栏工作台
  - 顶栏 / 页脚闭环
  - 右栏密度
  - rail 设备感
- 同时，**没有看到深度复刻把核心功能结构打断**。
- 当前离参考站最近的一步，主要还差：
  - 默认主题气质
  - hero 区继续瘦身
  - 控制栏去组件化
  - 空态 CTA 再收敛

### 一句话判断
- **部署：比上一轮一致很多，但还存在 2 个高优先级误导点。**
- **UI：已经进入“高相似 clone”区间，但还没到“默认打开第一眼就像参考站”的程度。**
